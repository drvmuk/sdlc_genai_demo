**Functional Requirements Document (FRD)**

**Project Title:** End-to-End ETL Pipeline for Product Data

**1. Requirement ID:** FR-BRZ-001

**2. Title:** Ingest Customer Data to Bronze Layer

**3. Description:** Ingest raw Customer data from CSV files located in a Unity Catalog Volume into a Delta table in the Bronze layer, applying SCD Type 2 principles.

**4. Actor(s):** DLT Pipeline, Auto Loader

**5. Preconditions:**

*   The DLT pipeline is running and configured to access the Unity Catalog.
*   The Unity Catalog Volume is accessible.
*   CSV files containing Customer data exist in the specified location (`/Volumes/catalog_sdlc/rawdata/customer`).

**6. Trigger:** New or updated CSV files are detected in the source directory by the Auto Loader.  Pipeline rerun is manually triggered.

**7. Main Flow / Functional Steps:**

1.  The DLT pipeline, using Auto Loader (cloud\_files with format="csv"), monitors the specified Customer data path (`/Volumes/catalog_sdlc/rawdata/customer`) for new or updated CSV files.
2.  Upon detecting a new or updated file, Auto Loader reads the CSV data.
3.  The data is ingested into the `bronzezone.data.customer_raw` Delta table.
4.  SCD Type 2 is applied to the Customer data based on the `id` column as primary key.
5.  The system keeps history of changes.
6.  Watermark columns are maintained.
7.  `CreateDateTime`, `UpdateDateTime`, and `IsActive` columns are created.
8.  When updates arrive, the `UpdateDateTime` and `IsActive` flags are updated accordingly.

**8. Alternate Flows / Exceptions:**

*   If a file cannot be accessed or is corrupt, the pipeline should log the error and continue processing other files (if any).
*   If the schema of the input CSV file does not match the expected schema, the pipeline should log the schema mismatch and potentially stop, depending on configuration (Schema Evolution is enabled, but use with caution).

**9. Postconditions / Outputs:**

*   The `bronzezone.data.customer_raw` Delta table is updated with the new or changed Customer data, including historical data and SCD Type 2 attributes.
*   Metadata for the ingested data is updated in the Unity Catalog.

**10. Business Rules:**

*   SCD Type 2 is applied using `id` as the primary key.
*   CreateDateTime, UpdateDateTime, and IsActive columns must be maintained for each record.

**11. Data Requirements (Optional):**

*   Input: CSV file(s) with columns: `id(STRING)`, `name(STRING)`, `email(STRING)`, `age(INT)`.
*   Output: Delta table `bronzezone.data.customer_raw` with the same columns, plus `CreateDateTime`, `UpdateDateTime`, `IsActive`, and potentially watermark columns.

**12. Assumptions (Optional):**

*   The input CSV files are properly formatted and encoded.
*   The `id` column is unique and can be used as the primary key for SCD Type 2.

**13. Dependencies (Optional):**

*   Databricks Runtime with Delta Live Tables enabled.
*   Access to Unity Catalog and the specified volume.

**14. Priority:** High

**15. Acceptance Criteria:**

*   All Customer data from the CSV files is successfully ingested into the `bronzezone.data.customer_raw` table.
*   SCD Type 2 is correctly implemented, preserving historical data.
*   `CreateDateTime`, `UpdateDateTime`, and `IsActive` columns are populated accurately.
*   The pipeline handles new and updated files incrementally.

**1. Requirement ID:** FR-SLV-001

**2. Title:** Load and Combine Customer and Order Data into Silver Layer

**3. Description:** Load data from the Bronze layer tables (`customer_raw` and `orders_raw`), join them, cleanse the data, and load into a Silver layer Delta table, applying SCD Type 2.

**4. Actor(s):** DLT Pipeline

**5. Preconditions:**

*   The DLT pipeline is running and configured to access the Unity Catalog.
*   The `bronzezone.data.customer_raw` and `bronzezone.data.orders_raw` Delta tables are populated with data.

**6. Trigger:** New or updated data is available in the `bronzezone.data.customer_raw` or `bronzezone.data.orders_raw` Delta tables. Pipeline rerun is manually triggered.

**7. Main Flow / Functional Steps:**

1.  The DLT pipeline reads data from the `bronzezone.data.customer_raw` and `bronzezone.data.orders_raw` Delta tables using Auto Loader (cloud\_files).
2.  The two datasets are joined on the `id` column.
3.  Records with any NULL values are removed.
4.  Duplicate records are removed.
5.  SCD Type 2 is applied using `id` as the primary key.
6.  CreateDateTime, UpdateDateTime, and IsActive flag are created.
7. Update logic updates UpdateDateTime and IsActive when new data arrives.
8.  The resulting data is loaded into the `silverzone.data.customer_order_combined` Delta table.

**8. Alternate Flows / Exceptions:**

*   If the join operation fails, log the error and potentially stop the pipeline.
*   If there are no matching `id` values during the join, log the number of unmatched records from each table.

**9. Postconditions / Outputs:**

*   The `silverzone.data.customer_order_combined` Delta table is populated with cleansed and combined data from the Customer and Order tables, including SCD Type 2 history.

**10. Business Rules:**

*   Data is joined on the `id` column.
*   Records with NULL values are excluded.
*   SCD Type 2 is applied using `id` as the primary key.

**11. Data Requirements (Optional):**

*   Input: `bronzezone.data.customer_raw`, `bronzezone.data.orders_raw`.
*   Output: `silverzone.data.customer_order_combined` (Delta Table).

**12. Assumptions (Optional):**

*   The `id` column is consistent between the Customer and Order tables.

**13. Dependencies (Optional):**

*   `bronzezone.data.customer_raw`
*   `bronzezone.data.orders_raw`

**14. Priority:** High

**15. Acceptance Criteria:**

*   The `silverzone.data.customer_order_combined` table contains a combined dataset from the two bronze tables.
*   No records with NULL values exist in the table.
*   Duplicate records are removed.
*   SCD Type 2 is correctly implemented.

**1. Requirement ID:** FR-GLD-001

**2. Title:** Load and Aggregate Data into Gold Layer

**3. Description:** Load data from the Silver layer (`silverzone.data.customer_order_combined`), perform aggregations, and load into a Gold layer Delta table (`goldzone.data.customer_order_summary`), applying SCD Type 2.

**4. Actor(s):** DLT Pipeline

**5. Preconditions:**

*   The DLT pipeline is running and configured to access the Unity Catalog.
*   The `silverzone.data.customer_order_combined` Delta table is populated with data.

**6. Trigger:** New or updated data is available in the `silverzone.data.customer_order_combined` Delta table. Pipeline rerun is manually triggered.

**7. Main Flow / Functional Steps:**

1.  The DLT pipeline reads data from the `silverzone.data.customer_order_combined` Delta table using Auto Loader (cloud\_files).
2.  The data is grouped by age or email domain.
3.  The following metrics are aggregated:
    *   Total revenue
    *   Average order amount
4. SCD Type 2 is applied to the aggregated data based on Group by column as primary key.
5. CreateDateTime, UpdateDateTime, and IsActive flag are created.
6. Update logic updates UpdateDateTime and IsActive when new data arrives.
7.  The resulting aggregated data is loaded into the `goldzone.data.customer_order_summary` Delta table.

**8. Alternate Flows / Exceptions:**

*   If the aggregation operation fails, log the error and potentially stop the pipeline.

**9. Postconditions / Outputs:**

*   The `goldzone.data.customer_order_summary` Delta table is populated with aggregated customer and order data, including SCD Type 2 history.

**10. Business Rules:**

*   Data is grouped by age or email domain.
*   Total revenue and average order amount are calculated for each group.
*   SCD Type 2 is applied using Group by column as the primary key.

**11. Data Requirements (Optional):**

*   Input: `silverzone.data.customer_order_combined`.
*   Output: `goldzone.data.customer_order_summary` (Delta Table).

**12. Assumptions (Optional):**

*   The grouping columns (age and email domain) are suitable for aggregation and analysis.

**13. Dependencies (Optional):**

*   `silverzone.data.customer_order_combined`

**14. Priority:** High

**15. Acceptance Criteria:**

*   The `goldzone.data.customer_order_summary` table contains aggregated data, grouped by either age or email domain.
*   Total revenue and average order amount are calculated correctly for each group.
*   SCD Type 2 is correctly implemented.
