**1. Requirement ID:** FR-BRZ-001

**2. Title:** Ingest Customer Raw Data into Bronze Layer

**3. Description:** This function ingests customer data in CSV format from a Unity Catalog Volume into the bronze layer of the data pipeline, applying SCD Type 2 logic.

**4. Actor(s):** Auto Loader, DLT Pipeline, Unity Catalog.

**5. Preconditions:**
    *   A Unity Catalog Volume containing customer data in CSV format exists.
    *   The DLT pipeline is configured and running.
    *   The `bronzezone.data.customer_raw` table does not need to exist initially; the helper function in `utils/catalog_helper.py` should handle creation.
**6. Trigger:** New CSV files become "available" in the configured Unity Catalog Volume location.  Trigger mode is continuous streaming with `trigger = "availableNow"` or `"once"` (determined by metadata inference).

**7. Main Flow / Functional Steps:**

    *   The Auto Loader monitors the specified Unity Catalog Volume for new CSV files.
    *   When new files are detected, the `cloud_files` function reads the CSV data.
    *   The data is loaded into a PySpark DataFrame within the `src/bronze/load_customer_raw.py` script using the `@dlt.table` decorator.
    *   SCD Type 2 logic is applied to the customer data using the `id` field as the primary key. This logic will create new records when there are changes, while preserving historical data.
    *   The following columns are managed as part of the SCD Type 2 implementation: `CreateDateTime`, `UpdateDateTime`, and `IsActive`.
    *   The resulting data is written to the `bronzezone.data.customer_raw` table in Unity Catalog.
    *   Checkpointing and incremental loads are enabled to ensure efficient processing of new data.

**8. Alternate Flows / Exceptions:**

    *   If the Unity Catalog Volume is inaccessible, the DLT pipeline will log an error and attempt to retry.
    *   If the CSV data is malformed or does not conform to the expected schema, the DLT pipeline will log an error and may quarantine the file.
    *   If there are issues writing to the `bronzezone.data.customer_raw` table, the DLT pipeline will log an error and attempt to retry.

**9. Postconditions / Outputs:**

    *   Customer data is ingested into the `bronzezone.data.customer_raw` table in Unity Catalog.
    *   SCD Type 2 history is maintained for customer data.

**10. Business Rules:**

    *   The `id` field is the primary key for SCD Type 2 updates.
    *   `CreateDateTime`, `UpdateDateTime`, and `IsActive` columns must be correctly maintained for SCD Type 2.

**11. Data Requirements (Optional):**

    *   Input: CSV data from Unity Catalog Volume (schema details will depend on the customer data itself).
    *   Output: `bronzezone.data.customer_raw` table in Unity Catalog.

**12. Assumptions (Optional):**

    *   The Unity Catalog is configured correctly and accessible.
    *   The CSV data in the Unity Catalog Volume conforms to a defined schema.

**13. Dependencies (Optional):**

    *   `cloud_files` function.
    *   `utils/scd2.py` (SCD Type 2 logic function).
    *   `utils/catalog_helper.py` (Unity Catalog creation/check helper).

**14. Priority:** High

**15. Acceptance Criteria:**

    *   Customer data is successfully ingested from the Unity Catalog Volume into the `bronzezone.data.customer_raw` table.
    *   SCD Type 2 history is correctly maintained for updates to customer data.
    *   The pipeline handles malformed CSV data gracefully, logging errors as needed.
    *   The `bronzezone.data.customer_raw` table is created if it does not already exist.

**1. Requirement ID:** FR-SLV-001

**2. Title:** Transform Customer and Orders Data into Silver Layer

**3. Description:** This function joins customer and order data from the bronze layer, removes nulls and duplicates, and applies SCD Type 2 logic to create a combined customer order table in the silver layer.

**4. Actor(s):** DLT Pipeline.

**5. Preconditions:**

    *   The `bronzezone.data.customer_raw` and `bronzezone.data.orders_raw` tables exist and contain data.
    *   The DLT pipeline is configured and running.
    *   The `silverzone.data.customer_order_combined` table does not need to exist initially; the helper function in `utils/catalog_helper.py` should handle creation.

**6. Trigger:** Data updates in either the `bronzezone.data.customer_raw` or `bronzezone.data.orders_raw` tables.

**7. Main Flow / Functional Steps:**

    *   The `src/silver/transform_customer_orders.py` script using the `@dlt.table` decorator reads data from the `bronzezone.data.customer_raw` and `bronzezone.data.orders_raw` tables.
    *   The customer and order data are joined on the `id` field.
    *   Null values are removed from the joined dataset.
    *   Duplicate records are removed using SHA256 hash-based deduplication.
    *   SCD Type 2 logic is applied to the combined data.
    *   The resulting data is written to the `silverzone.data.customer_order_combined` table in Unity Catalog.

**8. Alternate Flows / Exceptions:**

    *   If either the `bronzezone.data.customer_raw` or `bronzezone.data.orders_raw` table is inaccessible, the DLT pipeline will log an error and attempt to retry.
    *   If the join operation fails, the DLT pipeline will log an error.
    *   If there are issues writing to the `silverzone.data.customer_order_combined` table, the DLT pipeline will log an error and attempt to retry.

**9. Postconditions / Outputs:**

    *   A combined customer and order dataset is available in the `silverzone.data.customer_order_combined` table in Unity Catalog.
    *   Null values and duplicates are removed.
    *   SCD Type 2 history is maintained for the combined data.

**10. Business Rules:**

    *   The `id` field is used for joining the customer and order data.
    *   SHA256 hash-based deduplication is used for removing duplicates.
    *   SCD Type 2 history is maintained.

**11. Data Requirements (Optional):**

    *   Input: `bronzezone.data.customer_raw` and `bronzezone.data.orders_raw` tables.
    *   Output: `silverzone.data.customer_order_combined` table.

**12. Assumptions (Optional):**

    *   The `bronzezone.data.customer_raw` and `bronzezone.data.orders_raw` tables have a compatible schema for joining on the `id` field.

**13. Dependencies (Optional):**

    *   `utils/scd2.py` (SCD Type 2 logic function).
    *   `utils/catalog_helper.py` (Unity Catalog creation/check helper).

**14. Priority:** High

**15. Acceptance Criteria:**

    *   Data is successfully joined from the bronze layer and loaded into the `silverzone.data.customer_order_combined` table.
    *   Null values and duplicates are correctly removed.
    *   SCD Type 2 history is correctly maintained.
    *   The `silverzone.data.customer_order_combined` table is created if it does not already exist.

**1. Requirement ID:** FR-SEC-001

**2. Title:** Access Control (ACL) Implementation

**3. Description:** This function provides code snippets to grant `SELECT` and `MODIFY` access to specified roles for Unity Catalog tables.

**4. Actor(s):**  Administrator, Unity Catalog

**5. Preconditions:**

    *   The roles that will be granted access are already defined in Unity Catalog.
    *   The Unity Catalog tables (e.g., `bronzezone.data.customer_raw`, `silverzone.data.customer_order_combined`, `goldzone.data.customer_order_summary`) exist.
    *   An Administrator with sufficient privileges is available to execute the code snippets.

**6. Trigger:** A need arises to control access to Unity Catalog tables.  This could be after table creation or when a new role is introduced.

**7. Main Flow / Functional Steps:**

    *   An administrator identifies the Unity Catalog table and the roles to which access needs to be granted.
    *   The administrator retrieves the appropriate code snippet (e.g., SQL commands, Databricks CLI commands, or Python code) for granting `SELECT` and `MODIFY` privileges from documentation or configuration.
    *   The administrator executes the code snippet, substituting the correct table name and role names.
    *   Unity Catalog updates its permissions to reflect the granted access.

**8. Alternate Flows / Exceptions:**

    *   If the administrator does not have sufficient privileges, the access control update will fail, and an error will be logged.
    *   If the specified role does not exist, the access control update will fail, and an error will be logged.
    *   If the specified table does not exist, the access control update will fail, and an error will be logged.

**9. Postconditions / Outputs:**

    *   The specified roles have `SELECT` and/or `MODIFY` access to the designated Unity Catalog table.

**10. Business Rules:**

    *   The code snippets provided must be compliant with Unity Catalog's access control syntax and best practices.
    *   Separate roles should be created for read-only (SELECT) and read-write (MODIFY) access.

**11. Data Requirements (Optional):**

    *   Input: Table name (e.g., `bronzezone.data.customer_raw`), Role name (e.g., `data_scientists`, `analysts`), Privilege type (e.g., `SELECT`, `MODIFY`)

**12. Assumptions (Optional):**

    *   The Unity Catalog is properly configured.
    *   Administrators understand Unity Catalog's access control model.

**13. Dependencies (Optional):**

    *   Unity Catalog APIs or command-line interface.

**14. Priority:** Medium

**15. Acceptance Criteria:**

    *   The provided code snippets are syntactically correct and executable.
    *   The code snippets grant the specified roles the correct access levels to Unity Catalog tables.
    *   Verification that the roles can successfully `SELECT` and/or `MODIFY` the target tables.
