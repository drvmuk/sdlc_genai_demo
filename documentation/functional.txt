**1. Requirement ID:** FR-BRONZE-001

**2. Title:** Ingest Customer Data from Unity Catalog Volume

**3. Description:** Ingest customer data in CSV format from a Unity Catalog Volume into a Databricks Delta table within the bronze layer. Implement checkpointing and incremental loads for efficient data ingestion. Apply SCD Type 2 logic to maintain a history of changes.

**4. Actor(s):** Databricks DLT Pipeline, Unity Catalog Volume, Customer Data (CSV)

**5. Preconditions:**

*   The Unity Catalog Volume containing the customer data CSV files must exist.
*   The target Databricks Delta table (`bronzezone.data.customer_raw`) must be either non-existent (for initial load) or existing (for incremental loads).
*   Appropriate permissions must be in place to read from the Unity Catalog Volume and write to the Delta table.

**6. Trigger:** The DLT pipeline is triggered either manually or based on a defined schedule. New data appearing in the Unity Catalog Volume will also trigger the ingestion.

**7. Main Flow / Functional Steps:**

1.  The `load_customer_raw.py` script within the bronze layer is initiated.
2.  The script utilizes `cloud_files` to ingest CSV data from the specified Unity Catalog Volume.
3.  Checkpointing and incremental load mechanisms are enabled to process only new or updated files.
4.  SCD Type 2 logic is applied using the customer ID as the primary key, creating `CreateDateTime`, `UpdateDateTime`, and `IsActive` columns.
5.  The ingested data is written to the `bronzezone.data.customer_raw` Delta table.

**8. Alternate Flows / Exceptions:**

*   If a CSV file in the Unity Catalog Volume has an invalid format, the record will be skipped, and an error will be logged.
*   If the target Delta table does not exist, the script will create it.
*   If there are permission issues reading from the Unity Catalog Volume or writing to the Delta table, the DLT pipeline will fail.

**9. Postconditions / Outputs:**

*   The `bronzezone.data.customer_raw` Delta table is populated with the latest customer data from the Unity Catalog Volume.
*   A history of customer data changes is maintained in the Delta table through SCD Type 2.
*   Metadata about the load (e.g., number of records ingested, timestamp) is available in DLT pipeline logs.

**10. Business Rules:**

*   SCD Type 2 logic must be implemented to track changes to customer data over time.
*   The `IsActive` flag should be set to `True` for the latest version of a customer record and `False` for previous versions.

**11. Data Requirements:**

*   **Input:** CSV files containing customer data in the Unity Catalog Volume. The CSV files should include a customer ID column.
*   **Output:** `bronzezone.data.customer_raw` Delta table with columns: Customer ID, Customer Data Fields, `CreateDateTime`, `UpdateDateTime`, `IsActive`.

**12. Assumptions:**

*   The Unity Catalog Volume is accessible and properly configured.
*   The CSV files in the Unity Catalog Volume adhere to a consistent schema.

**13. Dependencies:**

*   PySpark
*   Databricks Delta Lake
*   Unity Catalog
*   `cloud_files`

**14. Priority:** High

**15. Acceptance Criteria:**

*   The `bronzezone.data.customer_raw` Delta table is successfully created or updated with data from the Unity Catalog Volume.
*   SCD Type 2 logic is correctly implemented, maintaining a history of customer data changes.
*   Checkpointing and incremental load mechanisms are functional.
*   Error handling is implemented to gracefully handle invalid data or permission issues.
*   Unit tests in `tests/test_bronze.py` pass successfully using dummy data and mocking of the Delta table.

**1. Requirement ID:** FR-SILVER-001

**2. Title:** Combine Customer and Order Data

**3. Description:** Join the `bronzezone.data.customer_raw` and `bronzezone.data.orders_raw` Delta tables on the common ID column. Remove nulls and duplicates using SHA256 hash-based deduplication. Apply SCD Type 2 upsert logic.

**4. Actor(s):** Databricks DLT Pipeline, `bronzezone.data.customer_raw` Delta table, `bronzezone.data.orders_raw` Delta table.

**5. Preconditions:**

*   The `bronzezone.data.customer_raw` and `bronzezone.data.orders_raw` Delta tables must exist and contain data.
*   The tables must have a common ID column for joining.
*   Appropriate permissions must be in place to read from the bronze layer tables and write to the silver layer table.

**6. Trigger:** The DLT pipeline is triggered either manually or based on a defined schedule, or when new data is available in either `bronzezone.data.customer_raw` or `bronzezone.data.orders_raw`.

**7. Main Flow / Functional Steps:**

1.  The `transform_customer_orders.py` script within the silver layer is initiated.
2.  The script reads data from the `bronzezone.data.customer_raw` and `bronzezone.data.orders_raw` Delta tables.
3.  The tables are joined based on the common ID column.
4.  Null values are removed from the joined data.
5.  Duplicates are removed using SHA256 hash-based deduplication.
6.  SCD Type 2 upsert logic is applied.
7.  The transformed data is written to the `silverzone.data.customer_order_combined` Delta table.

**8. Alternate Flows / Exceptions:**

*   If either of the bronze layer Delta tables is missing, the DLT pipeline will fail.
*   If the join operation results in an empty dataset, the DLT pipeline will log a warning.
*   If the target Delta table does not exist, the script will create it.
*   If there are permission issues, the DLT pipeline will fail.

**9. Postconditions / Outputs:**

*   The `silverzone.data.customer_order_combined` Delta table is populated with combined, cleaned, and deduplicated customer and order data.
*   SCD Type 2 history is maintained.

**10. Business Rules:**

*   Data deduplication must be implemented to ensure data quality.
*   SCD Type 2 logic must be correctly applied.

**11. Data Requirements:**

*   **Input:** `bronzezone.data.customer_raw` and `bronzezone.data.orders_raw` Delta tables.
*   **Output:** `silverzone.data.customer_order_combined` Delta table with columns from both input tables, as well as `CreateDateTime`, `UpdateDateTime`, and `IsActive`.

**12. Assumptions:**

*   The common ID column exists in both input Delta tables.
*   The `scd2.py` function is correctly implemented and available.

**13. Dependencies:**

*   PySpark
*   Databricks Delta Lake
*   `scd2.py`

**14. Priority:** High

**15. Acceptance Criteria:**

*   The `silverzone.data.customer_order_combined` Delta table is successfully created or updated.
*   The join operation is performed correctly.
*   Null values and duplicates are removed as expected.
*   SCD Type 2 upsert logic is correctly implemented.
*   Unit tests in `tests/test_silver.py` pass.

**4. Actor(s):** All DLT pipeline scripts (bronze, silver, gold).

**5. Preconditions:**

*   The Databricks environment is configured to use Unity Catalog.

**7. Main Flow / Functional Steps:**

1.  The DLT pipeline script calls the `catalog_helper.py` helper function, passing in the catalog, schema, and table name.
2.  The helper function checks if the specified catalog.schema.table exists in Unity Catalog.
3.  If the catalog.schema.table does not exist, the helper function creates it.
4.  The helper function returns (or raises an exception if creation fails).

**8. Alternate Flows / Exceptions:**

*   If there are permission issues when checking for or creating the catalog.schema.table, the helper function will raise an exception.

**9. Postconditions / Outputs:**

*   The target catalog.schema.table exists in Unity Catalog.

**10. Business Rules:**

*   The helper function should ensure that the catalog, schema, and table are created if they do not already exist.

**11. Data Requirements:**

*   **Input:** Catalog name, Schema name, Table name (string).
*   **Output:** None (but the table is created in Unity Catalog if it doesn't exist).

**12. Assumptions:**

*   The catalog and schema names are valid according to Unity Catalog naming conventions.
*   The user has sufficient permissions to create catalogs, schemas, and tables.

**13. Dependencies:**

*   PySpark
*   Databricks Unity Catalog

**14. Priority:** High

**15. Acceptance Criteria:**

*   The helper function successfully creates the catalog.schema.table if it doesn't exist.
*   The helper function does not attempt to re-create the catalog.schema.table if it already exists.
