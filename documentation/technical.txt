**Technical Requirement Document**

**1. Technical Requirement ID:** TR-BRZ-001

**2. Related Functional Requirement(s):** FR-BRZ-001

**3. Objective:** Implement a Databricks job using Delta Live Tables (DLT) and Auto Loader to ingest customer data from a Unity Catalog Volume into the bronze layer (`bronzezone.data.customer_raw` table), applying SCD Type 2 logic for historical data tracking.

**4. Target Cluster Configuration:**

    *   Cluster Name: dlt_pipeline_cluster
    *   Databricks Runtime Version: 13.3 LTS or higher (for DLT compatibility and cloud_files support)
    *   Node Type: Standard_DS3_v2 (General Purpose)
    *   Driver Node: 1
    *   Worker Nodes: 2-8 (Scalable based on data volume)
    *   Autoscaling: Enabled (Min Workers: 2, Max Workers: 8)
    *   Auto Termination: Enabled (30 minutes of inactivity)
    *   Libraries Installed:  `delta-sharing`

**5. Source Data Details:**

    *   Dataset Name: Customer Raw Data
    *   Location (Path/Table): Unity Catalog Volume (specified in DLT pipeline configuration)
    *   Format: CSV
    *   Description: Customer data in CSV format, including an `id` field for SCD Type 2, and other relevant customer attributes.

**6. Target Data Details:**

    *   Output Name: Customer Raw Bronze Table
    *   Location: `bronzezone.data.customer_raw` (Unity Catalog)
    *   Format: Delta
    *   Description: Bronze layer table containing raw customer data with SCD Type 2 history. Includes `CreateDateTime`, `UpdateDateTime`, and `IsActive` columns.

**7. Job Flow / Pipeline Stages:**

    *   **Stage 1: Data Ingestion (Auto Loader):** Auto Loader continuously monitors the specified Unity Catalog Volume for new CSV files.
    *   **Stage 2: Data Loading:**  New CSV files are read using the `cloud_files` function.
    *   **Stage 3: SCD Type 2 Application:** The data is loaded into a PySpark DataFrame within the `src/bronze/load_customer_raw.py` script using the `@dlt.table` decorator. SCD Type 2 logic (from `utils/scd2.py`) is applied based on the `id` field.  `CreateDateTime`, `UpdateDateTime`, and `IsActive` columns are maintained.
    *   **Stage 4: Data Writing:** The resulting DataFrame is written to the `bronzezone.data.customer_raw` Delta table in Unity Catalog.

**8. Data Transformations / Business Logic:**

    *   **Step 1: CSV Parsing:** The `cloud_files` function parses the CSV files into a PySpark DataFrame.
    *   **Step 2: Schema Inference/Enforcement:**  The schema is either inferred by Auto Loader or explicitly defined in the DLT pipeline configuration.
    *   **Step 3: SCD Type 2 Logic:** The `utils/scd2.py` function is used to implement SCD Type 2 logic. The logic determines if a new record should be inserted (when the `id` is new) or if an existing record should be updated (when the `id` exists and other attributes have changed).
    *   **Step 4: Metadata Column Management:**  `CreateDateTime` is set to the current timestamp for new records. `UpdateDateTime` is set to the current timestamp for updated records.  `IsActive` is set to `true` for the current record and `false` for the previous record when a change occurs.

**9. Error Handling and Logging:**

    *   **Inaccessible Unity Catalog Volume:** Log an error message and attempt to retry.
    *   **Malformed CSV Data:** Log an error message, and potentially quarantine the file (implementation depends on error handling strategy).
    *   **Write Errors:** Log an error message and attempt to retry.
    *   Use the Databricks logging framework (`logging` module in Python) to log events at different levels (INFO, WARN, ERROR).
    *   DLT automatically provides pipeline-level logging and metrics.

**10. Scheduling & Triggering:**

    *   Trigger: New files "available" in the Unity Catalog Volume.
    *   Trigger Mode: Continuous streaming (`trigger = "availableNow"`) or `"once"` (determined by metadata inference, configurable).
    *   The DLT pipeline continuously monitors the source location for new files.

**11. Security & Access Control:**

    *   Use Unity Catalog to manage access to the `bronzezone.data.customer_raw` table.
    *   Grant `SELECT` privileges to roles that need to read the data.
    *   Grant `MODIFY` privileges only to the DLT pipeline service principal or a dedicated data engineering role.
    *   Refer to FR-SEC-001 for code snippets to implement ACLs.

**12. Dependencies:**

    *   `cloud_files` function.
    *   `utils/scd2.py` (SCD Type 2 logic function).
    *   `utils/catalog_helper.py` (Unity Catalog creation/check helper).
    *   Delta Lake libraries.

**13. Assumptions:**

    *   The Unity Catalog is configured correctly and accessible.
    *   The DLT pipeline has the necessary permissions to read from the Unity Catalog Volume and write to the `bronzezone.data.customer_raw` table.
    *   The CSV data in the Unity Catalog Volume conforms to a defined schema or can be reliably inferred by Auto Loader.

**14. Acceptance Criteria:**

    *   Customer data is successfully ingested from the Unity Catalog Volume into the `bronzezone.data.customer_raw` table.
    *   SCD Type 2 history is correctly maintained for updates to customer data (verify `CreateDateTime`, `UpdateDateTime`, `IsActive` columns).
    *   The pipeline handles malformed CSV data gracefully (log errors, potential quarantine).
    *   The `bronzezone.data.customer_raw` table is created if it does not already exist.
    *   Performance is acceptable for the expected data volume.

**15. Notes / Implementation Suggestions:**

    *   Consider using schema evolution in Auto Loader to handle schema changes in the source CSV files.
    *   Implement alerting on error conditions (e.g., using Databricks alerts or external monitoring tools).
    *   Consider data quality checks to validate the ingested data.
