**Technical Requirements Document (TRD): End-to-End ETL Pipeline for Product Data**

*   **1. Technical Requirement ID:** TR-BRZ-001
*   **2. Related Functional Requirement(s):** FR-BRZ-001
*   **3. Objective:** Implement a Delta Live Tables pipeline to ingest customer data from CSV files located in a Unity Catalog Volume into a Bronze layer Delta table, applying SCD type-2 logic and generating metadata columns (`CreateDateTime`, `UpdateDateTime`, `IsActive`).

*   **4. Target Cluster Configuration**

    *   Cluster Name: dlt-cluster-customer-bronze
    *   Databricks Runtime Version: 13.3 LTS or higher (for DLT and Auto Loader compatibility)
    *   Node Type: Standard_DS3_v2
    *   Driver Node: Same as Worker
    *   Worker Nodes: 2
    *   Autoscaling: Enabled (min workers: 2, max workers: 4)
    *   Auto Termination: Enabled (after 120 minutes of inactivity)
    *   Libraries Installed: None (DLT and Auto Loader are built-in)

*   **5. Source Data Details**

    *   Dataset Name: Customer Raw Data
    *   Location (Path/Table): `/Volumes/catalog_sdlc/rawdata/customer`
    *   Format: CSV
    *   Description: Raw customer data in CSV format, containing fields like id, name, email, and age.

*   **6. Target Data Details**

    *   Output Name: Customer Raw Bronze Table
    *   Location: `bronzezone.data.customer_raw`
    *   Format: Delta
    *   Description: Delta table in the Bronze layer containing the ingested customer data, with SCD type-2 history and metadata columns.

*   **7. Job Flow / Pipeline Stages**

    1.  **Data Ingestion:**  Auto Loader reads new or updated CSV files from the Unity Catalog Volume.
    2.  **Data Loading and Transformation:** The data is loaded into a temporary Delta table. SCD Type 2 logic is applied to merge the new data with the existing data in `bronzezone.data.customer_raw`. `CreateDateTime`, `UpdateDateTime`, and `IsActive` columns are populated.
    3.  **Table Update:**  The `bronzezone.data.customer_raw` table is updated with the transformed data.

*   **8. Data Transformations / Business Logic**

    *   Step 1: Ingest data from CSV using Auto Loader
        *   Description: Read CSV files from the configured path using `cloud_files` with `format="csv"`.
        *   Transformation Logic: `cloud_files("/Volumes/catalog_sdlc/rawdata/customer", "csv")`
    *   Step 2: Apply SCD Type 2 Logic
        *   Description: Merge incoming data with the existing Delta table, handling inserts, updates, and historical tracking.
        *   Transformation Logic:
            ```sql
            MERGE INTO bronzezone.data.customer_raw AS target
            USING updates AS source
            ON target.id = source.id
            WHEN MATCHED AND target.IsActive = true AND (target.name != source.name OR target.email != source.email OR target.age != source.age) THEN
              UPDATE SET target.IsActive = false, target.UpdateDateTime = current_timestamp()
            WHEN NOT MATCHED THEN
              INSERT (id, name, email, age, CreateDateTime, UpdateDateTime, IsActive)
              VALUES (source.id, source.name, source.email, source.age, current_timestamp(), current_timestamp(), true)
            ```

*   **9. Error Handling and Logging**

    *   Invalid data files: Log the error and the file name to a dedicated error log table in the `logging` catalog. Halt processing for that file. Send an alert to the Data Engineering Team via email or Slack.
    *   Table creation failure: Log the error and halt the pipeline. Send an alert to the Data Engineering Team.
    *   Use DLT expectations to validate data quality.

*   **10. Scheduling & Triggering**

    *   Schedule: The DLT pipeline will be triggered every 15 minutes.
    *   Triggering: The pipeline will also be triggered when new data is detected in the source CSV file location by Auto Loader.
    *   Delta Live Tables manages dependencies and execution order.

*   **11. Security & Access Control**

    *   Unity Catalog will be used to manage access to the source and target data.
    *   The DLT pipeline will run under a service principal with appropriate read/write permissions to the Unity Catalog Volume and Delta tables.

*   **12. Dependencies**

    *   Delta Live Tables.
    *   Unity Catalog.
    *   Auto Loader.

*   **13. Assumptions**

    *   The CSV file adheres to the specified schema (id: STRING, name: STRING, email: STRING, age: INT).
    *   The `id` field is a unique identifier for each customer and used as the primary key for SCD Type 2.

*   **14. Acceptance Criteria**

    *   Data is ingested correctly into the `bronzezone.data.customer_raw` table from the CSV file.
    *   SCD type-2 history is maintained accurately.
    *   `CreateDateTime`, `UpdateDateTime`, and `IsActive` columns are populated correctly.
    *   Invalid data files are handled gracefully, and errors are logged and alerted.
    *   Incremental data loading processes only new or changed records.

*   **15. Notes / Implementation Suggestions**

    *   Consider using Delta Live Tables expectations to enforce data quality rules.
    *   Explore using a configuration table to store pipeline parameters, such as the source file path and the target table name.

*   **2. Related Functional Requirement(s):** FR-SLV-001
*   **3. Objective:** Implement a Delta Live Tables pipeline to load, join, and cleanse customer and order data from the Bronze layer into a Silver layer Delta table (`silverzone.data.customer_order_combined`), applying SCD type-2 logic and generating metadata columns.

*   **4. Target Cluster Configuration**

    *   Cluster Name: dlt-cluster-silver
    *   Databricks Runtime Version: 13.3 LTS or higher (for DLT and Auto Loader compatibility)
    *   Node Type: Standard_DS3_v2
    *   Driver Node: Same as Worker
    *   Worker Nodes: 2
    *   Autoscaling: Enabled (min workers: 2, max workers: 4)
    *   Auto Termination: Enabled (after 120 minutes of inactivity)
    *   Libraries Installed: None (DLT and Auto Loader are built-in)

*   **5. Source Data Details**

    *   Dataset Name: Customer Raw Bronze Table
    *   Location (Path/Table): `bronzezone.data.customer_raw`
    *   Format: Delta
    *   Description: Customer data from the Bronze layer with SCD type-2 history.
    *   Dataset Name: Order Raw Bronze Table
    *   Location (Path/Table): `bronzezone.data.orders_raw`
    *   Format: Delta
    *   Description: Order data from the Bronze layer with SCD type-2 history.

*   **6. Target Data Details**

    *   Output Name: Customer Order Combined Silver Table
    *   Location: `silverzone.data.customer_order_combined`
    *   Format: Delta
    *   Description: Delta table in the Silver layer containing the combined and cleansed customer and order data, with SCD type-2 history and metadata columns.

*   **7. Job Flow / Pipeline Stages**

    1.  **Data Loading:** Read the latest active records from `bronzezone.data.customer_raw` and `bronzezone.data.orders_raw`.
    2.  **Data Joining:** Join the two datasets on the `id` column.
    3.  **Data Cleansing:** Remove records with null values and duplicate records.
    4.  **SCD Type 2 Application:** Apply SCD Type 2 logic to the combined data.
    5.  **Table Update:**  The `silverzone.data.customer_order_combined` table is updated with the transformed data.

*   **8. Data Transformations / Business Logic**

    *   Step 1: Load Data from Bronze Tables
        *   Description: Read active records from the Bronze layer tables.
        *   Transformation Logic: `SELECT * FROM bronzezone.data.customer_raw WHERE IsActive = true` and `SELECT * FROM bronzezone.data.orders_raw WHERE IsActive = true`
    *   Step 2: Join Customer and Order Data
        *   Description: Join the two datasets on the `id` column.
        *   Transformation Logic:
            ```sql
            SELECT c.*, o.order_amount, o.order_date
            FROM customer_raw c
            INNER JOIN orders_raw o ON c.id = o.id
            ```
    *   Step 3: Remove Null Values
        *   Description: Filter out records with any null values.
        *   Transformation Logic: `WHERE c.id IS NOT NULL AND c.name IS NOT NULL AND c.email IS NOT NULL AND c.age IS NOT NULL AND o.order_amount IS NOT NULL AND o.order_date IS NOT NULL`
    *   Step 4: Remove Duplicates
        *   Description: Deduplicate records.
        *   Transformation Logic: Use a window function like `ROW_NUMBER()` to assign a unique rank to each row and then filter for rows with rank = 1.
    *   Step 5: Apply SCD Type 2 Logic
        *   Description: Merge incoming data with the existing Delta table, handling inserts, updates, and historical tracking.
        *   Transformation Logic:
            ```sql
            MERGE INTO silverzone.data.customer_order_combined AS target
            USING updates AS source
            ON target.id = source.id
            WHEN MATCHED AND target.IsActive = true AND (target.name != source.name OR target.email != source.email OR target.age != source.age OR target.order_amount != source.order_amount OR target.order_date != source.order_date) THEN
              UPDATE SET target.IsActive = false, target.UpdateDateTime = current_timestamp()
            WHEN NOT MATCHED THEN
              INSERT (id, name, email, age, order_amount, order_date, CreateDateTime, UpdateDateTime, IsActive)
              VALUES (source.id, source.name, source.email, source.age, source.order_amount, source.order_date, current_timestamp(), current_timestamp(), true)
            ```

*   **9. Error Handling and Logging**

    *   Source tables unavailable: Log the error and halt the pipeline. Send an alert to the Data Engineering Team.
    *   Join failure: Log the error and halt the pipeline.
    *   Data quality issues (after joining): Log any data quality issues to a dedicated log table.
    *   Use DLT expectations to validate data quality throughout the process.

*   **10. Scheduling & Triggering**

    *   Schedule: The DLT pipeline will be triggered every 30 minutes.
    *   Triggering: The pipeline will also be triggered when new data is available in either of the Bronze layer tables (determined by DLT dependencies).
    *   Delta Live Tables manages dependencies and execution order.

*   **11. Security & Access Control**

    *   Unity Catalog will be used to manage access to the source and target data.
    *   The DLT pipeline will run under a service principal with appropriate read/write permissions to the Unity Catalog Volume and Delta tables.

*   **12. Dependencies**

    *   Delta Live Tables.
    *   Unity Catalog.
    *   Auto Loader (indirectly, through Bronze layer tables).

*   **13. Assumptions**

    *   The `id` field is present in both source tables and is reliable for joining.
    *   The data in the Bronze layer tables conforms to the expected schema.

*   **14. Acceptance Criteria**

    *   Data is loaded correctly into the `silverzone.data.customer_order_combined` table, joined on the `id` column.
    *   Null values and duplicate records are removed.
    *   SCD type-2 history is maintained accurately.
    *   `CreateDateTime`, `UpdateDateTime`, and `IsActive` columns are populated correctly.
    *   Invalid data issues are handled gracefully, and errors are logged and alerted.
    *   Incremental data loading processes only new or changed records.

*   **15. Notes / Implementation Suggestions**

    *   Consider using Delta Live Tables expectations extensively to validate data quality at each stage.
    *   Implement monitoring to track the number of records processed and any errors encountered.

*   **1. Technical Requirement ID:** TR-GLD-001
*   **2. Related Functional Requirement(s):** FR-GLD-001
*   **3. Objective:** Implement a Delta Live Tables pipeline to aggregate and summarize customer order data from the Silver layer into a Gold layer Delta table (`goldzone.data.customer_order_summary`), applying SCD type-2 logic and generating metadata columns. The aggregation should be configurable to group by either age or email domain.

*   **4. Target Cluster Configuration**

    *   Cluster Name: dlt-cluster-gold
    *   Databricks Runtime Version: 13.3 LTS or higher (for DLT and Auto Loader compatibility)
    *   Node Type: Standard_DS3_v2
    *   Driver Node: Same as Worker
    *   Worker Nodes: 2
    *   Autoscaling: Enabled (min workers: 2, max workers: 4)
    *   Auto Termination: Enabled (after 120 minutes of inactivity)
    *   Libraries Installed: None (DLT and Auto Loader are built-in)

*   **5. Source Data Details**

    *   Dataset Name: Customer Order Combined Silver Table
    *   Location (Path/Table): `silverzone.data.customer_order_combined`
    *   Format: Delta
    *   Description: Combined and cleansed customer and order data from the Silver layer with SCD type-2 history.

*   **6. Target Data Details**

    *   Output Name: Customer Order Summary Gold Table
    *   Location: `goldzone.data.customer_order_summary`
    *   Format: Delta
    *   Description: Delta table in the Gold layer containing the aggregated and summarized customer order data, with SCD type-2 history and metadata columns.  Columns will include either age or email domain (depending on the configuration), total revenue, and average order amount.

*   **7. Job Flow / Pipeline Stages**

    1.  **Data Loading:** Read the latest active records from `silverzone.data.customer_order_combined`.
    2.  **Aggregation:** Group the data by either `age` or `email domain` based on the configuration.
    3.  **Metric Calculation:** Calculate total revenue and average order amount for each group.
    4.  **SCD Type 2 Application:** Apply SCD Type 2 logic to the aggregated data.
    5.  **Table Update:**  The `goldzone.data.customer_order_summary` table is updated with the transformed data.

*   **8. Data Transformations / Business Logic**

    *   Step 1: Load Data from Silver Table
        *   Description: Read active records from the Silver layer table.
        *   Transformation Logic: `SELECT * FROM silverzone.data.customer_order_combined WHERE IsActive = true`
    *   Step 2: Determine Aggregation Criteria
        *   Description: Read the grouping criteria (age or email domain) from a configuration table or parameter.
        *   Transformation Logic: `SELECT grouping_criteria FROM config.pipeline_config WHERE pipeline_name = 'gold_aggregation'` (Example using a configuration table)
    *   Step 3: Aggregate Data
        *   Description: Group the data based on the configured criteria and calculate aggregate metrics.
        *   Transformation Logic:
            ```sql
            SELECT
              CASE
                WHEN (SELECT grouping_criteria FROM config.pipeline_config WHERE pipeline_name = 'gold_aggregation') = 'age' THEN age
                ELSE SUBSTRING(email, INSTR(email, '@') + 1)  -- Extract email domain
              END AS grouping_key,
              SUM(order_amount) AS total_revenue,
              AVG(order_amount) AS average_order_amount
            FROM silver_data
            GROUP BY grouping_key
            ```
    *   Step 4: Apply SCD Type 2 Logic
        *   Description: Merge incoming data with the existing Delta table, handling inserts, updates, and historical tracking.
        *   Transformation Logic: (Adapt based on grouping criteria, e.g., if grouping by age, the join condition is `target.age = source.age`)
            ```sql
            MERGE INTO goldzone.data.customer_order_summary AS target
            USING updates AS source
            ON target.grouping_key = source.grouping_key  --  Replace grouping_key with actual column name (age or domain)
            WHEN MATCHED AND target.IsActive = true AND (target.total_revenue != source.total_revenue OR target.average_order_amount != source.average_order_amount) THEN
              UPDATE SET target.IsActive = false, target.UpdateDateTime = current_timestamp()
            WHEN NOT MATCHED THEN
              INSERT (grouping_key, total_revenue, average_order_amount, CreateDateTime, UpdateDateTime, IsActive)  --  Replace grouping_key with actual column name (age or domain)
              VALUES (source.grouping_key, source.total_revenue, source.average_order_amount, current_timestamp(), current_timestamp(), true)
            ```

*   **9. Error Handling and Logging**

    *   Source table unavailable: Log the error and halt the pipeline. Send an alert to the Data Engineering Team.
    *   Aggregation failure: Log the error and halt the pipeline.
    *   Invalid configuration: Log an error and use a default aggregation (e.g., by age).
    *   Use DLT expectations to validate data quality throughout the process.

*   **10. Scheduling & Triggering**

    *   Schedule: The DLT pipeline will be triggered every hour.
    *   Triggering: The pipeline will also be triggered when new data is available in the Silver layer table (determined by DLT dependencies).
    *   Delta Live Tables manages dependencies and execution order.

*   **11. Security & Access Control**

    *   Unity Catalog will be used to manage access to the source and target data.
    *   The DLT pipeline will run under a service principal with appropriate read/write permissions to the Unity Catalog Volume and Delta tables.

*   **12. Dependencies**

    *   Delta Live Tables.
    *   Unity Catalog.
    *   Auto Loader (indirectly, through Silver layer table).
    *   (Optional) Configuration table for grouping criteria.

*   **13. Assumptions**

    *   The data in the Silver layer table conforms to the expected schema.
    *   If a configuration table is used, it exists and contains the required configuration.

*   **14. Acceptance Criteria**

    *   Data is loaded correctly into the `goldzone.data.customer_order_summary` table, aggregated by the specified criteria (age or email domain).
    *   Total revenue and average order amount are calculated accurately.
    *   SCD type-2 history is maintained accurately.
    *   `CreateDateTime`, `UpdateDateTime`, and `IsActive` columns are populated correctly.
    *   Invalid data issues are handled gracefully, and errors are logged and alerted.
    *   Incremental data loading processes only new or changed records.

*   **15. Notes / Implementation Suggestions**

    *   Implement dynamic configuration loading to allow for easy changes to the aggregation criteria.
    *   Consider window functions for future enhancements, such as calculating running totals or period-over-period comparisons.
    *   Monitor the performance of the aggregation queries, especially with large datasets.
